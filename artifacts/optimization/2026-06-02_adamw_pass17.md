# ft-optim AdamW Pass 17

Bead: `frankentorch-7pf5`

## Target

Fresh `ft-optim` optimizer target generated after ready perf beads were drained.
`AdamW::step` allocates per-parameter update buffers on the hot optimizer step
path. The existing Criterion harness exercises 64 parameter tensors with 1024
values each:

```text
cargo bench -p ft-optim --bench optimizer_bench -- adamw/step_64x1024 --warm-up-time 1 --measurement-time 5 --sample-size 20
```

## Baseline

```text
worker: vmi1156319
adamw/step_64x1024 time: [990.15 us 1.0232 ms 1.0572 ms]
```

## Lever

Reuse the `param_values` buffer fetched from the session as the outgoing values
buffer in `AdamW::step`, mutating each element in place before
`tensor_update_param_values`. This removes the per-parameter `update` allocation
and the second pass through `apply_param_update` for AdamW while leaving the
generic optimizer helper unchanged.

Clippy also required two crate-local mechanical cleanups before the final
`ft-optim` gate could pass: `numel.is_multiple_of(cols)` and a direct array
slice in a test.

## Isomorphism Proof

- Parameter order unchanged: the loop still iterates `self.params` in order.
- Gradient loading and missing-gradient skip behavior unchanged.
- First and second moment formulas unchanged; the code still materializes
  `next_m` and `next_v` before mutating parameter values.
- Floating-point order per element unchanged: compute `m_hat`, `v_hat`,
  `adam_delta`, `decay_delta`, then subtract `decay_delta + adam_delta`.
- Step-count mutation remains after successful parameter update.
- RNG and tie-breaking unaffected; AdamW has no RNG path.
